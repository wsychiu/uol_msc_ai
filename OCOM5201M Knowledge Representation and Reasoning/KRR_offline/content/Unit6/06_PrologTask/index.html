    Modelling modal logic in Prolog

    Learning Outcomes

    In this unit you will :

    • Learn some more advanced Prolog constructs.
    • Build your own implementation to evaluate modal logic formulas.
    • See how tense logic works as a special case of modal logic.
    • See how Prolog terms can be manipulated in Prolog to check the syntax of formulas in your implementation.

    Overview

    This unit provides experience in writing Prolog and also in understanding some of the concepts around relations and modal logic. It should help develop your understanding of description logic and will introduce you to tense logic.

    The unit will guide you through a series of stages in which you are given a number of tasks. To do this some basic Prolog programming is introduced first.

    1. Prolog ideas you will need

    We have not seen much Prolog so far. The examples here will help you with the task later. Consult online documentation if more details are needed, but there is enough here to explain what you need later.

    Remember that names starting with a capital letter are always variables.

    1.1 Comma in body of clause

    To say:

    • If X is green and X is a monster then X is a green monster.

    we can write this clause:

     greenMonster(X) :- green(X), monster(X).

    You should test it works by adding suitable facts like:

     green(thing1).
 green(thing2).
 green(thing3).
 monster(thing2).
 monster(thing3).
 monster(thing4).

    That means you add those to the database of facts and clauses. If you are using SWISH Prolog this will be on the left-hand side. You can run queries such as these (one at a time) to see what happens.

     greenMonster(thing2).

 greenMonster(thing4).

 greenMonster(thing42).

 greenMonster(M).

    In the last case you can see more than one answer by selecting 'next' in SWISH.

    1.2 Multiple clauses for same predicate

    We often need to write multiple clauses with the same head and different bodies. To say:

    • Things which are green are dangerous and things that are monsters are dangerous

    we can put:

      dangerous(X) :- green(X).
  dangerous(X) :- monster(X).

    Try this out with the earlier facts.

    1.3 Getting output

    Sometimes writing things out is helpful, especially in debugging code. Experiment with this example:

      greenMonWrite(X) :- write(X),
                      writeln("  is a green monster").

  testMon(X) :- greenMonster(X), greenMonWrite(X).

    1.4 The built-in predicates findall and foreach

    To write out all the green monsters the foreach built-in predicate is valuable:

       foreach(greenMonster(M), greenMonWrite(M)).

    or you can define:

       writeAllTheGreenMonsters :- foreach(greenMonster(M), greenMonWrite(M)).

    Do something similar with finding all the dangerous things and writing them out. You will get some twice because there is more than one reason why some are dangerous.

    Try this to get a list of all the dangerous things:

       findall(X, dangerous(X),ListOfThem).

    You will find:

        ListOfThem = [thing1, thing2, thing3, thing2, thing3, thing4]

    1.5 List processing

    Prolog can deal with lists, which are always enclosed in square brackets and with the elements separated by commas. Lists can contain other lists as elements, and the empty list, [ ], is often seen, especially as the base case in recursively defined predicates.

    What if we do not want the duplicates in the list of all dangerous things?

         removeDups([], []).

     removeDups([FirstOne | Rest ], RestDone) :- member(FirstOne, Rest),
                                                 removeDups(Rest, RestDone).

     removeDups([FirstOne | Rest ], [FirstOne | RestDone]) :-
                                                 removeDups(Rest, RestDone).

    This is a very common structure, processing one list into another. - The built in predicate member tests to see if something is a member of a list.

    • A list with one or more elements will match a pattern [H|T] by taking:

      • H to be the first element of the list
      • T to be the list (possibly empty) that results by removing H from the list.

    Try these goals to see what happens:

          [H|T] = [1,2,3].
      [H|T] = [1].
      [H|T] = [ ].
      [H|T] = [[1],2,3].
      [H|T] = [1,[]].
      [H|T] = [[ ]].
      [H|[H2|TT]] = [1,2,3].
      [H|[H2|[TT]]] = [1,2,3].

    1.6 Anonymous variables

    Suppose we have a list of just two things and we want to know if there is a dangerous thing there.

    One way is simply:

          beware([X,Y]) :- dangerous(X).
      beware([X,Y]) :- dangerous(Y).

    This works (try it to see) but produces a warning, and it is better to write:

         beware([X,_]) :- dangerous(X).
     beware([_,Y]) :- dangerous(Y).

    with an anonymous variable, the underscore. Beware that if the underscore occurs several times in a single clause or goal, they can all be successfully matched with different things.

    1.7 Using the cut, !, to prevent backtracking

    The pattern in the following includes the exclamation mark ! which is called cut. This means that if X is green then the second clause is never tried when backtracking occurs:

         checkGreen(X) :- green(X), !.
     checkGreen(X) :- write("Warning: "),
                      write(X), writeln(" is not green"),
                      fail.

    You can experience the effect of the cut by trying this goal

         checkGreen(thing1),monster(thing1).

    with two scenarios: with and without the ! in the first clause.

    1.8 Equals

    If you try this as a goal

     X = a.

    it succeeds with X instantiated to a. You can use this to unify expressions, but Prolog does not check for cyclic cases. The following succeeds but can be confusing:

     X = p(X).

    The following is easy to understand; try it as a goal.

    p(a,Y) = p(X,b).

    If you try this as a goal

       X == a.

    it fails because the Prolog terms are not the same.

    2. Modelling modal logic in Prolog

    In this part you will write a Prolog program that is capable of finding whether a given expression is true at a state in a given interpretation. The description logic we saw in unit 4 is a special kind of modal logic, and we will see how to model time later in this unit.

    2.1 Setting up states and relations

    We start with four states:

    state(fido).
state(rover).
state(felix).
state(wally).

    We can have a language with propositions that hold for some states:

    holds(dog, fido).
holds(dog, rover).
holds(cat, felix).
holds(cat, wally).

    The next ingredient we need is some relations between the states. We can do this:

    rel(chases, felix, fido).
rel(chases, fido,  rover).
rel(likes,  wally, fido).
rel(older,  wally, fido).

    These mean that the chases relation holds between Felix and Rover in that order. If we drew a picture there could be an arrow labelled "chases" going from Felix to Fido.

    So far we have set up a structure which can be seen as an interpretation for a modal logic language in which there are:

    • Propositions: \(\mathit{dog}\), and \(\mathit{cat}\).
    • The usual logical connectives \(\vee\), \(\wedge\), \(\neg\), \(\to\).

    • Boxes and diamonds:

      • \([\mathit{chases}]\), \([\mathit{likes}]\),\([\mathit{older}]\)
      • \(\langle \mathit{chases} \rangle\), \(\langle \mathit{likes} \rangle\), \(\langle \mathit{older} \rangle\).

    We are giving a valuation in which: \(\mathit{dog}\) is true at the states fido and rover, and in which \(\mathit{cat}\) is true at the states felix and wally.

    In our interpretation we have provided three actual binary relations on the set of all states. These are going to be needed to interpret the modal operators.

    To test the pieces that you are going to write, you may need to add to or change the states, propositions and relations as you experiment.

    2.2 The propositional connectives

    Your task here is to write clauses so that the holds predicate will handle expressions involving the logical connectives. The clauses must be able to deal with complex logical expressions such as:

      implies( and( not(implies(dog,not(cat))), or(not(dog), or(bird,hippo))), thing)

    but of course you need to work out each connective separately with one or more clauses for the holds predicate.

    The following goals should all succeed:

       holds( and(dog, not(cat)), fido).

   holds( or(cat, dog), wally).

   holds( not(dog), felix).

   holds( implies(dog, cat), wally).

    The following should all fail:

      holds( and(dog, cat), fido).

  holds( or(not(cat), dog), wally).

  holds( not(not(dog)), felix).

  holds( implies(cat, dog), wally).

    To test thoroughly you may need to introduce some more atomic propositions and give the states where they hold. You may also need to introduce more states. Make sure all the connectives work exactly as the truth tables say they should.

    Your definitions must work whatever the states and atomic propositions are.

    2.3 holding everywhere or validity

    Write predicates:

    holdsEverywhere(Formula)

holdsSomewhere(Formula)

    which succeed if and only if the formula is true at all states or at some state respectively.

    You should find that these succeed:

     holdsEverwhere( or(cat, not(cat)) ).

 holdsSomewhere( cat ).

    2.4 Adding the diamond and the box

    Now extend the language so we can have formulas like these:

    diamond( Relation, Formula)

box( Relation, Formula)

    You need to extend the holds predicate so that these succeed:

    holds( diamond( chases, dog ), felix).

holds( box( older, dog ), wally).

    These say that:

    • Felix chases some dog.
    • Wally is only older than dogs.

    are both true. Beware that this is saying they are true in this particular interpretation. This is quite different from saying they must be true in all interpretations, which would be the effect of stating these as assertions in description logic.

    To test thoroughly you will need to add some more states, propositions and relations. Don't forget nesting is allowed, so your rules have to work correctly with formulas like:

    diamond( chases, box( older, dog ) )

    and possibly including all the earlier connectives.

    2.5 Operations on relations

    So far we have used only the explicitly given relations. We have met operations of inverse and composition of roles in description logic. So we extend the language to allow not just relation names but also these expressions:

    compose(Rel1, Rel2)

inverse(Rel)

    You need to write clauses for the rel predicate to make this work. The following goals would succeed (given the initially stated facts):

    rel(compose(likes, chases), wally, rover).

rel(inverse(likes), fido, wally).

    These would be saying:

    • Wally likes someone who chases Rover.
    • Fido is liked by Wally.

    Again, be careful that what we are doing is describing a particular interpretation. We are not setting out axioms that are required to hold in all cases.

    You should be able to include further operations on relations so that expressions can include:

    union(Rel1, Rel2)

intersect(Rel1, Rel2)

not(Rel)

    To be clear what these mean:

    • Two things are related in the union of relations if those two things are related in at least one of the relations.
    • Two things are related in the union of relations if those two things are related in both relations.
    • Two things are related in the negation of a relation, \(R\), if those two things are not related in \(R\).

    Make sure you test the operations on relations carefully with suitable examples, and check they work when you use then in logical expressions. Goals of the following form should work correctly:

    holds( diamond( compose(likes,chases), dog), wally).

    in other words:

    • Wally likes someone who chases a dog.

    2.6 properties of relations

    In Protégé we saw the significance of the seven characteristics that roles may have. You should now write predicates as follows:

      reflexive(R)

  transitive(R)

  symmetric(R)

  functional(R)

  invfun(R)

  irrefl(R)

  asymm(R)

    That for a relation R will succeed just when it has the associated property.

    3 Time

    In tense logic there four tense operators:

    • \(\mathbf{F} \varphi\) meaning \(\varphi\) is true at some time in the future.
    • \(\mathbf{P} \varphi\) meaning \(\varphi\) is true at some time in the past.
    • \(\mathbf{G} \varphi\) meaning \(\varphi\) is true at every time in the future.
    • \(\mathbf{H} \varphi\) meaning \(\varphi\) is true at every time in the past.

    We are going to have to use a finite number of times here so we can check all the times. The logic itself is not limited in this way.

    If we have ten times, we can set up an interpretation of tense logic as follows and use our earlier Prolog predicates to evaluate expressions.

      state(t0).
  state(t1).
  state(t2).
  state(t3).
  state(t4).
  state(t5).
  state(t6).
  state(t7).
  state(t8).
  state(t9).

    and then add a relation from each time to every future time. This would be painful, so we can do this:

      state(t(X)) :- 0 =< X, X =< 9.

  rel(before, t(X), t(Y)) :- X < Y.

    You should now be able to use the earlier box, diamond, and inverse relation ideas to define things to make these work.

      holds(future(X), S)

  holds(past(X), S)

  holds(goingToBe(X), S)

  holds(hasAlwaysBeen(X), S)

    To test your definitions and your understanding, how would you express the following in tense logic (using suitable atomic propositions)?

    • I have never been to Iceland.
    • To win the lottery you must have bought a ticket.
    • Drinking alcohol and driving are each legal separately, but not at the same time.
    • Every time I was in Manchester it was raining.

    Test your formulas in your implementation.

    4 Checking the language syntax

    This part may be a bit more challenging, but can be completed without knowing any more Prolog than the ideas introduced earlier and some new ones below.

    We can try this query:

    holds(not(hippo), john).

    This will succeed when there is no state john and no predicate hippo.

    So we cannot distinguish failure from having the value 'false' and failure from not being valid expression.

    Extend your program so that when given a goal such as the above you get a useful error message such as:

     hippo is not a proposition
 john is not a state

    but the holds predicate should work as before on acceptable expressions.

    You can extend this so that it checks the logical elements too. For example it could detect and explain the syntax errors in these.

    holds(and(cat,dog,cat), wally).

holds(nonsense(cat,dog), fido).

holds(diamond(cat,cat), wally).

holds(fido, dog).

    You may find the following built-in predicates useful:

         atom(X). % is X an atom?

     functor(T, F, A). % succeeds if T is a term, with functor F
                       % having arity A. The case of arity zero is allowed.

     not(fred(bill), cat) =.. X.
                       % will produce  = [not, fred(bill), cat]

    For Prolog the predicate atom will succeed when its argument is hippo but will fail if it is a variable or a complex term. See these goals for examples to try out:

         atom(hippo).

     atom(and(dog, cat)).

     atom(and).

     atom(Something).
