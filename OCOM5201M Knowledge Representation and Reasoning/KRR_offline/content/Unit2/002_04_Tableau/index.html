    Lesson 4: The tableau method

    Learning Outcomes

    After completing this lesson you should be able to:

    • Use the tableau method to determine if a formula \(\varphi\) is satisfiable, and, if so, find a model for it.
    • Use the tableau method to determine whether a claimed logical implication \(\alpha_1, \ldots, \alpha_n \vDash \beta\) holds.
    • Explain the key capabilities and limitations of the tableau method.

    You will learn first how to do these for propositional logic, and then for first order logic.

    Overview of key features

    • The tableau method is a method for determining logical entailment
    • Versions of the technique work for propositional logic, first order logic, as well as description logics as used in the Semantic Web, and other modal logics, etc, etc
    • It's a refutation method: to show \(\Gamma \vDash \alpha\) we show \(\Gamma \cup \{\neg \alpha\}\) unsatisfiable.
    • Suitable for implementation
    • Gives Decision Procedure for propositional logic and Semi-Decision Procedure for first order logic

    The third stage has four nodes aligned vertically but the fourth node branches downwards to the left and right into two separate nodes. The first and seconds nodes have green ticks beside them.

    The fourth stage has four nodes aligned vertically. The fourth node branches off into a fifth and sixth node, both of which branch off again to make a further four nodes. The first, second and third nodes have green ticks beside them.

    The fifth stage has four nodes aligned vertically, all of which have green ticks beside them. The fourth node branches off to nodes on the left- and right-hand side. Secondary branches split off from these nodes. Each of these has three further vertically aligned nodes growing downwards, forming four more branches of nodes in total. The nodes at the bottom of the second and third of these branches has a red cross underneath.

    Nodes are labelled by formulas of the logic. At each stage nodes are processed to generate new nodes (formulas). The tree may branch as a result of this processing. When a node has been used as much as possible to generate new nodes a tick, \(\checkmark\), (shown green in the diagram) is added to denote this. When it is impossible to extend a path any further it is said to be closed and this is shown in the diagram by a red cross (\(\times\)).

    We are interested in paths from the root to leaves. Each path, as it grows, is an attempt to find a way to make all the formulas on the path satisfiable. The attempts to do this proceed by reducing the task of making complicated formulas satisfiable to sub-tasks of making simpler formulas satisfiable. If a path gets closed, this means that particular attempt to make all the formulas on it satisfiable has failed. If all paths get closed then there is no way to make all the formulas we started with satisfiable. If some path is open (not closed) but we cannot develop the path any further then we have found a way to make all the formulas we started with satisfiable.

    In the case of first order logic, but not in propositional logic, it is possible that the process of growing the tree does not terminate.

    The Tableau method for first order logic

    Overview

    • The basic idea is the same as in the propositional case:

      • construct a tree by processing nodes to add new parts to the tree,
      • If all paths become closed then the initial sentences are unsatisfiable.

    • For first order logic there are two new rules: for \(\forall\) and for \(\exists\).

    • Nodes of the form \(\forall x \; \Phi\) never get marked as finished with — they can continue adding new parts to the tree and this can lead to infinite trees.

    • If the initial sentences are unsatisfiable then the process terminates with all paths closed. But a satisfiable set of sentences may lead to a tree that grows indefinitely.

    The existential quantifier rule

    Suppose we have a node labelled \(\exists x \, \Phi\) where \(x\) is free in \(\Phi\), and suppose that the node is on an open path containing constants \(c_1, c_2, \ldots, c_n\).

    Then we:

    1. add a new node labelled by \(\Phi [ x / c_{n+1} ]\) where \(c_{n+1}\) is a new constant not in \(\{c_1, c_2, \ldots, c_n\}\), and

    2. mark the node labelled \(\exists x \, \Phi\) as used.

    Note that a node of the form \(\neg \exists x \Phi\) generates a new one labelled \(\forall x \neg \Phi\) which is then dealt with by the \(\forall\) rule (see later).

    Example for \(\exists\) rule

    Determine whether \(\vDash \exists x P(x,a) \rightarrow \forall x P(a,x)\) (where \(a\) is a constant).

    This means the sentence \({\neg (\exists x P(x,a) \rightarrow \forall x P(a, x))}\) is satisfiable. The path shows one way to satisfy the sentence. We can have the domain \(\{1, 2, 3\}\) and interpret the constants \(a, b, c\) as \(1, 2, 3\) respectively, and interpret \(P\) as \(\{(2,1)\}\).

    The universal quantifier rule

    Suppose we have a node labelled \(\forall x \, \Phi\) where \(x\) is free in \(\Phi\), and suppose that the node is on an open path containing constants \(c_1, c_2, \ldots, c_n\). If there are no constants on the path then we introduce a new one (ok because domain must be non-empty).

    Then we:

    • add every one of the sentences \(\Phi [x/c_i]\) for \(i = 1, 2, \ldots, n\) to the end of the path (unless it is already on the path).

    Note that a node of the form \(\neg \forall x \Phi\) generates a new one labelled \(\exists x \neg \Phi\) which is then dealt with by the \(\exists\) rule (see earlier).
